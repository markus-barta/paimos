# Title: Unify per-user "Projects" and "Access" actions into a single matrix dialog

## Description

The user row in `SettingsUsersTab.vue` currently exposes two action
buttons that operate on the same backing table (`project_members`):

- **Projects** (`SettingsUsersTab.vue:318`, externals only) — opens a
  curated list of grants with an Add picker. Hardcodes `viewer`. Backed
  by legacy `/api/users/{id}/projects`.
- **Access** (`SettingsUsersTab.vue:319`, all non-self users) — opens
  an N-row matrix with `none` / `viewer` / `editor` per project plus
  reset-to-default. Backed by `/api/users/{id}/memberships`.

The matrix is a strict superset of the legacy flow: anything the
Projects modal can do, the matrix can already do (and more). The two
UIs cause real admin confusion on external rows — it is unclear which
button is canonical, and the two paths produce two separate audit
trails for the same action.

The legacy UI's one true advantage is shape: external users start with
zero seeded rows, so for them the "list of grants" is short and
curated, while the full matrix shows ~30 rows of "None" noise. Staff
(admin/member) have the inverse problem and the matrix already suits
them well.

This ticket consolidates onto the matrix and teaches it to behave well
for both audiences via a role-aware default view.

## Acceptance Criteria

- [ ] The "Projects" button is removed from the user row in
      `SettingsUsersTab.vue`. Only "Access" remains for non-self users.
- [ ] The Access dialog gains a **"Show only explicit grants"** toggle.
      Default ON for `role === 'external'`, default OFF for
      `admin` / `member`. Toggle state is local to the dialog (not
      persisted); flipping it instantly re-renders the row set.
- [ ] When the toggle is ON, only rows where `access_level !==
      role-default` are listed. When OFF, all non-deleted projects are
      listed.
- [ ] Rows are sorted: explicit grants first (alphabetical by project
      key), defaulted rows after a subtle separator/divider. Sort
      applies regardless of toggle state.
- [ ] An **"Add project"** affordance sits at the top of the dialog:
      a project picker (excluding projects already in the visible
      list) plus a level dropdown (Viewer / Editor — Editor hidden
      for externals). Clicking Add issues the same
      `PUT /memberships/{projectId}` mutation and the new row appears
      at the top of the granted section.
- [ ] For external users, the **Editor** button in the per-row level
      group is hidden (not just disabled). External users can only be
      None or Viewer. Reset for an external returns to None.
- [ ] All access state mutations go through
      `/api/users/{id}/memberships` endpoints. The legacy
      `openUserProjects` / `addUserProject` / `removeUserProject`
      handlers and their state (`userProjectsTarget`, `userProjects`,
      `addProjectId`, `allProjects`, `projectsLoading`,
      `UserProjectAccess` interface) are removed from
      `SettingsUsersTab.vue`. The legacy `Project Access:` modal
      block (lines 459–480) is removed.
- [ ] The dialog title reads `Access: {username}`. Subtitle / hint
      text adapts: for externals it explains "Externals start with no
      access; assign per project." For staff it explains "Members
      default to editor; use this to override per project."
- [ ] Visual quality: the dialog feels designed, not a stitched-up
      table. The granted-section divider is subtle (single hairline
      with a tiny label like "Defaults"), the toggle is a proper
      segmented control or labelled switch, and the Add row reads as
      a single composed control rather than three loose inputs.
- [ ] No regressions in `npm run typecheck`, `npm test`, or
      `npm run build`. Existing `SettingsUsersTab` tests still pass.

## Out of Scope (Follow-ups)

- **Backend endpoint deprecation.** `GET/POST/DELETE
  /api/users/{id}/projects` and `handlers/user_projects.go` remain
  alive (and exercised by `backend/handlers/portal_test.go`). A
  follow-up ticket can retire them once this UI ships and any
  external scripts are confirmed unused.
- **Audit log consolidation.** The audit trail already flows through
  `RecordAccessChange()` from both endpoints, so no migration needed.

## Additional Requirements

- Touches:
  - `frontend/src/components/settings/SettingsUsersTab.vue`
  - possibly `frontend/src/components/settings/settings-shared.css`
    for the dialog + segmented control + divider styles.
- The role-default lookup needed for the "explicit grants only" filter
  already comes back on each `MembershipRow` (the matrix dialog
  already knows the default per row to render the Reset button).
- The matrix endpoints already accept `none` / `viewer` / `editor`,
  so no backend changes are required.
- External-specific UI rules (hide Editor, default toggle ON, copy)
  key off `membershipsTarget.role === 'external'`.
