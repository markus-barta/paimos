# Title: Project Manifest — tabbed editor (Manifest / Guardrails / Glossary) with "Structure with AI" button

## Description

`ProjectContextSection.vue` currently exposes a single textarea for the
Manifest JSON next to the Repos card. Two real workflow needs are not
served by that:

1. **Onboarding the manifest from prose.** New projects start with `{}`
   and an empty editor. Anyone with a paragraph of "we use Vue 3 + Go,
   deploy via deploy.sh, prod is fly.io, on-call is Markus" has to
   hand-shape that into the manifest schema before they can save.
2. **Two adjacent LLM-context surfaces are missing.** The team wants
   **Guardrails** (rules the LLM must obey when working on this project
   — e.g. "never run destructive git commands", "ADR-driven architecture
   changes only") and a **Glossary** (project-specific terms,
   acronyms, personas). They share the manifest's authoring shape — a
   structured JSON document scoped to one project — but each deserves
   its own dedicated editor and AI button.

The replacement is a **tabbed editor** in the right card of
`ProjectContextSection`: **Manifest**, **Guardrails**, **Glossary**.
Each tab is a JSON editor with the same Save flow, and each tab gets a
"**Structure with AI**" button that takes the textarea content (slop or
malformed JSON), POSTs it to the unified `/ai/action` dispatcher with
an action key per tab, and routes the structured-JSON result through
the existing `AiOptimizeOverlay` diff so the user accepts or rejects
before it lands in the editor.

Persistence: per scope discussion, **no new backend endpoint**.
Guardrails and Glossary live inside the existing
`project_manifests.data` JSON under reserved top-level keys
`_guardrails` and `_glossary`. The Manifest tab edits everything
**other than** those two reserved keys. Save merges the three slices
back into one document.

## Acceptance Criteria

- [ ] The right card in `ProjectContextSection.vue` exposes three tabs
      — **Manifest**, **Guardrails**, **Glossary** — using the same
      underline tab pattern as `SettingsView.vue` (so the visual idiom
      is already familiar).
- [ ] Switching tabs preserves unsaved drafts in the other tabs (per-
      tab draft state, not a single shared textarea).
- [ ] Manifest tab loads/saves via the existing
      `GET/PUT /projects/{id}/manifest` endpoints; on save it merges
      the user-edited manifest body with the live `_guardrails` and
      `_glossary` slices so neither is lost.
- [ ] Guardrails tab edits the `_guardrails` reserved key inside
      `manifest.data`. Save uses the same endpoint and merges in the
      live manifest body and `_glossary`.
- [ ] Glossary tab edits the `_glossary` reserved key. Same merge
      semantics as Guardrails.
- [ ] Each tab exposes a **"Structure with AI"** button next to Save.
      Clicking it POSTs to `/ai/action` with action keys
      `structure_manifest`, `structure_guardrails`,
      `structure_glossary` respectively, surfaces the candidate JSON
      through `AiOptimizeOverlay`, and on Accept replaces the tab's
      draft with the structured JSON (no immediate save — user still
      has to click Save).
- [ ] When the AI feature is unavailable
      (`useAiOptimize().available === false`), the AI button is
      disabled with a tooltip pointing to AI settings — same UX as
      every other AI button in the app.
- [ ] When the backend hasn't yet implemented one of the action
      handlers, the request fails gracefully (red error pill, the
      existing `errMsg` formatting) and the editor is left untouched.
- [ ] PAI-178 sentinel behavior is preserved: `populated` and
      `summary` events still fire correctly, where "has manifest"
      means there is at least one non-reserved key in `manifest.data`
      OR a non-empty `_guardrails` OR a non-empty `_glossary`.
- [ ] No regressions in `npm run typecheck`, `npm test`, or
      `npm run build`.

## Additional Requirements

- The reserved-key convention is documented in a short comment block
  at the top of the new component so future readers don't have to
  reverse-engineer it.
- Tab order matches the order users typically need them: Manifest
  first (most-edited), Guardrails second (set once per project),
  Glossary third (grows incrementally).
- The "Structure with AI" button uses the established `sparkles` icon
  + the `useAiOptimize` composable's `runRewriteAction` flow. No new
  composable, no new overlay component.
- Touches (frontend only):
  - `frontend/src/components/project/ProjectContextSection.vue`
    (split out the right card into a child component)
  - `frontend/src/components/project/ProjectManifestTabs.vue` (new)
  - `frontend/src/services/projectContext.ts` (add helpers to read/
    write reserved-key slices without round-tripping the whole doc
    through the parent)
- Backend follow-up (separate ticket, NOT in scope): add
  `structure_manifest`, `structure_guardrails`, `structure_glossary`
  action handlers under `backend/ai/` and register them in the
  dispatcher + `/api/ai/actions` catalog. Until those land the AI
  buttons render but error out at request time, which is the agreed
  trade-off.
